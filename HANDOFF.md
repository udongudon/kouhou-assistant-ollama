## 実装済みの機能

### コア機能

- [x] **2段階パイプライン**: 議案書テキスト → 中間サマリ JSON → チャネル別広報文
- [x] **入力方法2種**: テキスト貼り付け / ファイルアップロード（PDF・Word・テキスト）
- [x] **5チャネル生成**: LINE / X（5本セット）/ Facebook / Instagram / HP掲載文
- [x] **Ollama Cloud 対応**: `OLLAMA_API_KEY` で Ollama Cloud API に直接接続
- [x] **モデル選択 UI**: GPT-OSS 20B（デフォルト）/ GPT-OSS 120B / Gemma 4

### UI・UX

- [x] **文字数表示**: X は5投稿分の個別文字数（ハッシュタグ・URL除外）、他は合計文字数
- [x] **手動編集**: 各チャネルのテキストエリアで直接編集可能
- [x] **AI書き直し**: 「ここをこう直して」フォームで修正指示を入力 → 再生成
- [x] **修正履歴**: 修正前テキストを自動保存、過去版一覧から「この版に戻す」が可能
- [x] **白紙再生成ボタン**: 編集・履歴をすべてリセットして再生成
- [x] **DLボタン**: 各チャネルを `{channel}_{short_title}.txt` でダウンロード
- [x] **追加指示フォーム**: 全チャネル共通の追加指示（申込URLなど）を生成前に入力可能

### 認証・セキュリティ

- [x] **パスワード保護**: `APP_PASSWORD` 環境変数で簡易認証（未設定なら認証なし）
- [x] **個人情報フィルタ**: `prompts/extract.md` の抽出ルールで携帯番号等の個人連絡先を除外

### プロンプト設計

- [x] **中間サマリスキーマ**: `internal_logistics`（受付時刻・服装・署名者名・懇親会等）を含む詳細スキーマ
- [x] **LINE フォーマット**: 実際のLINE告知文を few-shot 実例として組み込み済み（フォーマット参考のみ。人名・内容は流用しない注記あり）
- [x] **X 5本セット**: 時系列（告知/1週間前/前日/当日/事後）で構造化、個別文字数カウント対応
- [x] **トランケート処理**: 議案書が3万字を超える場合は自動で切り詰め、警告表示

## 現在の問題点・注意事項

### 既知の動作上の問題

1. **GPT-OSS 20B での JSON 解析失敗（稀）**  
   `prompts/extract.md` への応答で稀にコードブロックなしの生 JSON が返らず解析に失敗する。`_parse_json_response()` に正規表現 fallback を実装済みだが、それでも失敗した場合は GPT-OSS 120B に切り替えると安定する。

2. **X 投稿の文字数カウントの精度**  
   `_render_char_counts()` でハッシュタグと URL を除外した概算文字数を表示しているが、X の実際のカウント（URLは23文字固定）とは厳密には一致しない。目安として使う前提。

3. **長い議案書の切り詰め**  
   `truncate_for_prompt()` で先頭3万字に切り詰めるため、事業概要が後半に記載されている議案書では情報が欠落することがある。切り詰めた場合はUIに警告が表示されるが、自動で「重要部分だけ抽出」する機能はない。

4. **手動編集と AI 書き直しの競合**  
   テキストエリアで手動編集した内容は `st.session_state["result_{channel_id}"]` に即時反映されるが、その後「白紙から再生成」を押すと手動編集がリセットされる（意図通りの動作だが、ユーザーに注意喚起するUIはない）。

### アーキテクチャ上の制約

- **Streamlit のセッション管理**: ブラウザリロードでセッションが失われる。永続化が必要な場合は `st.cache_resource` やファイル保存の仕組みが必要
- **シングルユーザー想定**: 複数人同時アクセス時はセッションが混在する可能性があるが、現状の委員会内での利用規模では問題ない想定

## 次に依頼すべき作業

### 1. 出欠回答URL挿入（優先度：高）

**背景**: LINE告知文でよく「LINEWORKSの出欠フォームURL」を末尾に入れたいというニーズがある。  
**依頼内容**: UI に「出欠回答URL（任意）」入力フィールドを追加し、入力があれば `prompts/line.md` の `●出欠回答` セクションに自動で挿入されるようにする。  
**実装箇所**: `app.py` の `_options_section()` に入力欄追加 → `generate.py` の `_build_user_payload()` に渡す → `prompts/line.md` に反映ルールを追記

### 2. 一括DL ZIP（優先度：高）

**背景**: 複数チャネルのテキストを個別にDLするのが手間。  
**依頼内容**: 全チャネルの生成済みテキストをまとめた ZIP ファイルをワンクリックでDLできるボタンを追加。  
**実装箇所**: `app.py` の `_render_results()` に `st.download_button` を追加。`io.BytesIO` + `zipfile.ZipFile` で実装可能。

### 3. プロンプト品質改善（優先度：中）

**背景**: 現状の GPT-OSS 20B デフォルトでは、特に Facebook・Instagram・HP の文体がやや単調になることがある。  
**依頼内容**:
- `prompts/facebook.md` / `prompts/instagram.md` / `prompts/website.md` に few-shot 実例を追加する
- 実例は広報委員会の担当者に「良かった投稿」を数件提供してもらい、プロンプトに組み込む

### 4. スキャンPDF対応（優先度：低）

**背景**: 過去の議案書がスキャンPDFで保存されているケースがある。  
**依頼内容**: `src/extract.py` に OCR ライブラリ（`pytesseract` または `google-cloud-vision`）を組み込む。  
**注意**: Tesseract の日本語対応は別途 `tessdata/jpn.traineddata` が必要。クラウドVision APIの方が品質は高い。
